Canvas 基础

正如NGUI有UIRoot, 那么UGUI就应该有Canvas。 该组件规定了游戏内玩家能看到的UI元素的区域, 在UnityEditor面板中, Canvas实际以一个GameObject节点绑定一个Canvas节点存在, 作为

显示的UI元素将作为子节点放置于Canvas下, 可视为Canvas仅管理当前Canvas下所有的子UI元素(包括子Canvas)

在不考虑sorting layer 和 order in layer的影响下, 当前Canvas管理的子节点的绘制顺序由层级面板的顺序决定, Canvas下第一个子节点将会第一个被绘制.

Canvas规定了三种渲染模式:

  • Screen Space - Overlay: 该渲染模式会将UI元素放置于场景之上的渲染屏幕上, 当屏幕大小或分辨率被修改后, Canvas会自动更改大小进行适配.

  • Screen Space - Camera: 应该是所了解到的用的最多的一个渲染模式. 该渲染模式下会将Canvas放置于指定相机的前给定的距离处, UI上所有的元素将由Canavas上绑定的相机进行渲染, 可以实现场景渲染和UI渲染分离. 在绑定相机的模式为正交模式时Canvas上的UI元素将以透视图进行渲染(从unity的官方文档上看这种应该是用于制作3DUI)

  • World Space: 该渲染模式下, Canvas和所管理的UI元素和场景对象并无区别, 我估计这种模式多用于VR项目..尤其是第一人称VR游戏

一般项目中, 所有的元素放置在同一个Canvas上就足够了, 但实际使用上会出现场景中存在多个Canvas的情况. 例如嵌套Canavs(处于优化的目的, 一个Canvas作为另一个Canvas的子节点, 并采用和父Canavas相同的渲染模式)

Canvas 进阶 : 排序标识sorting layer, sort order 和 order in layer

  • sort order

    Canvas默认为屏幕空间的2D对象, 当对象处于屏幕空间的时候, unity不会给对象分配sorting layer 和 order in layer属性, 而是采用sort order的属性进行排序.一般而言, sort order的值越低越先渲染. 当所有的sort order值都相同时, 渲染顺序为层级面板上物体的排列顺序 [该规则适用于其他仅处于屏幕空间的UI对象]

    例如

    1. 处于同一个Canvas下的不同的Canvas
    2. 当前Canvas的渲染模式为Screen Space - Overlay.

  • sorting layer & order in layer

    当Canvas变为世界空间的对象时, Canvas和Canvas管理的UI对象默认由对象的世界坐标的z值(深度)来决定. 离绑定的相机越远越先渲染. 某个离相机较远的元素后渲染, 可以通过设置sorting layer
    去确定排序(绘制顺序为Layers & Tags中sorting layer的定义顺序来决定, 排序越前越先渲染)

    而order in layer作为sorting layer的细分部分, 可以实现在不改变sorting layer的情况下修改order in layer的值去实现渲染排序。order in layer值越小越先渲染

    Tips:

    • Canvas下的子Canvas, 可以通过勾选Override Sorting选项去重写sorting layer 和 ordering layer的值
    • 对象处于世界空间下时, 渲染顺序的排序级别为: Camera的depth值 > sorting layer > order in layer

    这里使用脏标记模式的设计思想: 用一个标志位来标记对象数据的内容是否发生变化, 如果没有发生变化就直接使用缓存数据, 不需要计算. 如果标记已经被改变, 则需要新的原始数据计算目标数据

备注

  • 脏标记的设置: 这里的脏标记分为顶点脏标记和材质球脏标记.
    顶点的脏标记一般在UI元素的初始化或被回收时被调用, 或者触发OnRectTransformDimensionsChange事件时候会改变(一般在组件大小改变, 节点的父子关系发生变化, 移动设备上的屏幕方向发生改变时触发), 以及各种需要改变组件大小和展示内容的场合
    材质球的脏标记一般在UI元素的初始化或被回收时被调用, 有例外的就是Image或者Text的某些属性发生变化(比如图元或者字体集的改变);以及Mask相关组件的位置更改

  • Canvas的动静分离: 由于画布刷新系统的机制, 被设置为脏标记的UI元素的Canvas会重新计算布局信息和顶点信息之后提交渲染. 如果一些静态UI元素和变化的UI元素处于一个Canvas下, 就会发生
    多余的运算。因而在制作上更建议: 同一个界面下的UI, 可活动元素和静态元素放置在不同的Canvas下, 尽管该操作会打断动态合批, 但是减少网格重建时间

  • 过多的Canvas反而会打断合批, 增加Drawcall. 所以在使用上需要严格控制Canvas的数量
